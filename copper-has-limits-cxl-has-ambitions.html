Where Memory and Optics Intersect

Semiconductor experts have been saying “memory is the bottleneck” for long enough that most people following AI infrastructure already know it. Memory is the most expensive and most critical resource in the stack. But here is a question worth sitting with: how much of that expensive memory is actually being used?

According to Microsoft Azure research, up to 30% of DRAM installed in servers is stranded. It has been allocated but sits idle, and it cannot be handed off to another server. The single most expensive component in a server’s cost structure, wasted at roughly one in three units the moment it gets installed.

There is a technology designed to fix this.

CXL, Compute Express Link

CXL is an open interconnect standard that lets CPUs, GPUs, and accelerators connect to memory in a cache-coherent way. It takes memory out of its fixed socket and makes it something that can be dynamically allocated and reclaimed wherever it is needed.

In 2024, the prevailing view was that CXL was dead in the AI era. Then the second half of 2025 hit and the picture shifted fast. Microsoft Azure deployed CXL memory in production. A DRAM shortage drove both Google and NVIDIA to start seriously evaluating CXL as an alternative. The explosion of KV cache demand from LLM inference created a concrete use case for offloading to CXL memory.

By early 2026, the market started moving in earnest. For CXL to expand beyond a single server and scale across an entire datacenter, it has to break through the physical limits of copper. Optical interconnect technology has been advancing fast enough that a path through that wall is now opening up. Run CXL protocol over optics, and memory that was previously trapped inside a server can become a shared resource across the entire datacenter. Companies have already started placing billion-dollar bets on exactly that combination.

You alread know that memory is the bottleneck in AI infrastructure. You probably already know that optics is central to the next generation of interconnects. If you have not yet seen how CXL connects those two things, this article is where to start.

1. What CXL Actually Is

CXL is an interconnect standard that allows CPUs, GPUs, and accelerators to connect to memory in a cache-coherent way. It started as a CPU-centric spec, but from CXL 3.0 onward, GPUs and accelerators can access memory directly without going through the CPU, and devices can share memory with each other.

To understand why this matters, you need to look at how servers have traditionally been built.

In a conventional server, each CPU manages only the DRAM physically attached to its socket. You populate the DDR5 DIMM slots, and that is the server’s total memory. The GPU has its own HBM, the NIC has its own buffer, the SSD has its own controller. Each device’s memory is physically bound to that device.

The problem is the waste this creates. According to Microsoft Azure research, up to 30% of installed server DRAM is stranded. Memory that has been allocated but is not actually in use, and cannot be reassigned to another server. For a hyperscaler, this translates to billions of dollars of DRAM sitting in racks doing nothing.

CXL offers the most direct solution to this problem. There are still real challenges around scheduling, NUMA architecture, and software stack maturity, but at the protocol level, CXL is the only open standard that can take memory out of the server and make it shareable. It operates across three sub-protocols.

CXL.io handles device discovery, initialization, and management over PCIe. Think of it as the “new device detected” mechanism for CXL devices entering the system.

CXL.mem lets a CPU read and write to memory attached to an external device as if it were local DRAM. The key word is load/store: the CPU issues standard memory instructions and gets data back directly, without constructing network packets or waiting for acknowledgment. Latency is higher than local DRAM (tens to hundreds of nanoseconds), but the access model is the same.

CXL.cache lets external devices access CPU-side memory with cache coherency maintained. When a GPU or accelerator caches data from CPU memory, the synchronization with the CPU happens automatically.

For a non-engineer, the simplest way to put it:

CXL is a system where all the chips inside a server share one large memory space and stay in sync in real time about what each of them has modified.

That is what cache coherency means.

CXL devices fall into three types.

Type 1 accesses CPU memory coherently but has no memory of its own. SmartNICs are the classic example.

Type 2 has its own memory (HBM, GDDR) while also accessing CPU memory. GPUs and FPGAs belong here.

Type 3 provides memory for the CPU to map into its address space. This is the hottest category in the market right now. Memory expanders and memory poolers are all Type 3.

Why does Type 3 matter so much? Because in the AI era, memory is the most expensive, most scarce, and most wasted resource in the system.

2. The CXL Spec Timeline: 1.0 Through 4.0

Here is how the spec has evolved.

CXL 1.0 (2019): Intel-driven first release. Built the coherency protocol on top of PCIe 5.0 physical layer. Single-device connections only.

CXL 2.0 (November 2020): The first real game-changer. Switching was added. A single CXL switch could now connect multiple hosts to multiple devices. This is where memory pooling became possible; multiple CPUs sharing one memory pool.

CXL 3.0 (August 2022): PCIe 6.0 doubled bandwidth to 64 GT/s. The critical addition was fabric support. Where earlier CXL was a tree structure (root CPU, branch devices), 3.0 introduced Port-Based Routing (PBR). PBR assigns each device a unique address, and switches forward data to the right destination based on that address, similar to how a postal system works. This enabled flexible topologies like mesh and ring, support for up to 4,096 nodes, and peer-to-peer communication between devices without going through the CPU. Memory sharing was also added: where pooling means one user at a time, sharing means multiple hosts can view the same memory region simultaneously with real-time synchronization.

CXL 3.1 (November 2023): Extension of 3.0. Global Integrated Memory (GIM) enabled direct memory sharing between different servers. The Fabric Manager API was formalized for centralized PBR switch management. Security was strengthened with TSP, providing encrypted isolation for each tenant’s data in multi-tenant environments. The intent was clear: turn the entire datacenter into one unified memory system.

CXL 4.0 (November 18, 2025): The latest spec. PCIe 7.0-based at 128 GT/s, doubling bandwidth again. Three new features were introduced.

First, bundled ports. Multiple physical CXL ports are combined into a single logical connection. Think of it as widening a four-lane highway to sixteen lanes. This enables up to 1.5 TB/s bandwidth over a single connection.

Second, native x2 width. CXL connections use physical lanes; x16 means sixteen lanes, x4 means four. Previously, x2 was a fallback mode used only when errors occurred. Now it is a first-class operating mode. This matters because it improves fan-out, the number of devices a single switch can connect simultaneously. Instead of one sixteen-lane road, you can lay eight two-lane roads. More devices per switch.

Third, four retimer support , up from two. A retimer is a signal repeater. Electrical signals degrade as they travel through copper, and retimers read and re-transmit the signal cleanly. Four retimers substantially increases physical reach. This points directly toward multi-rack CXL configurations.

And there is one line in the spec document: “Optical support with PCIe 7.0.”

Official recognition of optical connectivity. The implications of that one line are covered in detail later.

3. Where CXL Actually Stands Today: A Reality Check

The vision is ambitious. But is CXL a core AI datacenter interconnect right now?

No, not yet.

In March 2024, SemiAnalysis published a piece titled “CXL is Dead in the AI Era.” The core argument: AI shifted hyperscaler priorities. Scale-up GPU interconnect is owned by NVLink and Google ICI. Scale-out networking belongs to Ethernet and InfiniBand. CXL has nowhere to fit.

There is a technical basis for this. CXL/PCIe and Ethernet are built on fundamentally different design philosophies. CXL/PCIe is a load/store memory access protocol. Because it deals directly with memory data, any error halts the system. The PHY layer therefore demands extremely low error rates, and Forward Error Correction (FEC) is used minimally to avoid adding latency. Ethernet is a packet-based network protocol that uses FEC aggressively to correct errors, extracting higher bandwidth efficiency from the same physical channel. The result is that the same number of pins delivers more data over Ethernet. SemiAnalysis pointed out this tradeoff: chip I/O pins are a finite resource, and allocating them to CXL/PCIe at lower bandwidth efficiency is a meaningful cost.

So is CXL dead? Also no. The situation has shifted since 2024.

First, Microsoft Azure has started real deployments. In November 2025, Microsoft announced a CXL memory preview deployment in Azure M-series VMs using Astera Labs’ Leo CXL Smart Memory Controller. The first CSP to do so. Not yet GA, but the world’s largest cloud operator putting real money behind CXL is a signal.

Second, the DRAM shortage brought CXL back into the conversation. The race to build AI infrastructure has made DRAM supply extremely tight. The Information reported that both Google and NVIDIA are actively evaluating CXL as an alternative memory technology. When DRAM efficiency directly affects AI infrastructure economics, CXL’s memory pooling becomes a practical solution rather than a theoretical one.

Third, the KV cache problem blew up. KV cache is the data AI models store in memory to remember what has been said in a conversation. As conversations get longer (as context windows grow), this data grows explosively. A 70B parameter model with 128K context and batch size 32 requires more than 150GB of KV cache, exceeding the 80GB VRAM on an H100. This is exactly where CXL memory creates an opening. Offloading KV cache from GPU VRAM to CXL pooled memory can achieve 4-5x cost reduction versus VRAM and 200-500x lower latency versus SSD, according to various analyses. Marvell’s own testing showed a 4.8x improvement in inference throughput and 82.7% reduction in time to first token with CXL pooling. These numbers come from Marvell’s optimized test environment, and real-world production results will vary by workload.

The positioning is this. CXL is not a primary scale-up interconnect for AI clusters. It is not competing with NVLink. CXL’s role is as a supplementary tool for memory tier expansion and efficiency. But the economic value of that supplementary role runs into the billions, and that makes it impossible to ignore. The CXL memory expansion market is projected to grow from roughly $1.3B in 2025 to $11.8B by 2034 (MarketIntelo, 28.7% CAGR).

Estimates vary across research firms, but the direction is consistent.

4. Hyperscaler CXL Strategy: Google and Microsoft

Google is a founding member of the CXL Consortium and one of the most pragmatic adopters of the technology. Industry observers note that Google has been working with Marvell on custom CXL solutions. At Google’s scale, datacenters hold large volumes of retired DDR4 memory from server refresh cycles. The scenario of using CXL memory expanders to repurpose that DDR4 as cold tier memory (a slower, cheaper memory layer for data that is not frequently accessed) is reportedly under discussion at both Google and Oracle.

Marvell’s Structera X 2404 targets exactly this scenario. It accepts up to twelve DDR4 128GB modules pulled from retired servers, delivering up to 1.5TB of physical capacity. Apply LZ4 compression at line rate, and effective capacity rises to roughly 3TB. Expanding memory capacity without buying new DDR5 translates to hundreds of millions of dollars in annual DRAM procurement savings at hyperscaler scale.

The intersection of Google’s TPU ecosystem and CXL is also worth watching. Google’s seventh-generation TPU, Ironwood, has been announced, and Anthropic has a contract giving it access to up to one million TPUs through its Google Cloud partnership. At that scale, memory efficiency is a central challenge, and CXL memory pooling is a natural part of the solution.

Microsoft has already been covered with the Azure M-series VM deployment of Astera Labs Leo. The goal is to push past the DIMM slot limits on CPU sockets for memory-intensive workloads like in-memory databases. A server with sixteen DDR5 slots maxes out at 16TB with 1TB RDIMMs. CXL memory expanders break through that ceiling.

This is where CXL stands today. The technology is well-defined, the market is opening, and the hyperscalers have started committing capital. But there is one fundamental physical constraint CXL has not yet solved. Copper. And optical solutions are emerging as the next step beyond that constraint.

5. The Copper Wall: Why CXL Cannot Fly

CXL’s vision calls for fabrics spanning thousands of nodes, memory pools that cross rack boundaries, and fully disaggregated datacenters. The problem is that this vision runs straight into the physics of copper wiring.

As data transfer speeds increase, signal attenuation over copper gets dramatically worse. Electrical signals convert to heat as they travel through copper, and the faster the signal, the more severe the effect. At PCIe 5.0 (32 GT/s), copper cables had an effective reach of around 3 meters. At PCIe 6.0 (64 GT/s), that dropped below 1 meter. PCIe 7.0 (128 GT/s) is shorter still.

Lightelligence data makes this vivid. Send a PCIe 5.0 signal 10 meters over standard AWG26 copper cable (0.4mm diameter), and signal strength drops to one ten-thousandth of the original (-40 dB). At that point, recovering the original data on the receiving end is essentially impossible. Send that same signal over optical fiber? Signal strength drops by 0.07% (-0.003 dB). Nearly lossless.

The physical bulk difference is equally dramatic. Achieving comparable bandwidth over copper requires thirty-two cables at 6mm diameter or larger. Optical fiber needs just sixteen strands at 0.125mm, thinner than a strand of hair. Factor in cable weight, routing space inside server racks, and cooling overhead, and the gap widens further.

PBR enables CXL 3.1 to target thousands of nodes in theory. But real AI clusters are already linking tens of thousands of GPUs together. NVIDIA’s vision of million-GPU clusters is physically impossible with copper-based CXL. One server, maybe an adjacent rack on a good day. That is the practical reach of copper CXL.

There is a more fundamental issue. CXL is a protocol that lives and dies on nanosecond latency. Local DRAM access takes around 100ns. A pooled CXL memory access through a switch runs around 140-160ns by Marvell’s estimates. Maintaining that kind of latency while extending physical reach means the signal retiming over copper becomes a burden.

Can you use Ethernet instead? No. CXL is a protocol for directly reading and writing memory. Ethernet and RDMA are network protocols that package data into packets and send them. The latency gap is not marginal. CXL runs around 100ns; RDMA over Ethernet runs around 3 microseconds. That is a 30x difference. In memory transactions, 3 microseconds is an eternity.

The conclusion: for CXL to expand beyond the server and reach pod scale or rack scale, optics is the only viable path.

6. Optical CXL: Memory on Light

The concept of optical CXL began taking concrete shape in 2023.

A company called Lightelligence announced a product called Photowave, a transceiver that converts PCIe/CXL signals to optical and sends them over fiber. It was the first commercial optical CXL hardware. The product came in three form factors (PCIe card, OCP 3.0 SFF card, and AOC), with CXL 2.0/PCIe Gen 5 support and bifurcation mode for flexible deployment scenarios.

At Flash Memory Summit 2023, Lightelligence ran a live demo alongside AMD, MemVerge, and Samsung, demonstrating LLM inference on an AMD Genoa server connected to a Samsung CXL memory expander over a Photowave optical link. It was a proof of concept showing that standard CXL protocol could run over fiber.

Why does this matter? Copper CXL confines the memory expander to within the server, physically within centimeters of the CPU. Optical CXL extends that to tens or hundreds of meters. This is an architectural shift, not just a cabling upgrade.

Once CXL memory is no longer trapped inside the server, it can be pooled across racks and pods, across the entire datacenter. Compute racks and memory shelves sit physically separate, connected by fiber. The CPU issues load/store instructions to memory that is meters away, using the exact same access model as local DRAM, with higher latency but nothing like what a network hop would cost. This is the foundational structure of a disaggregated datacenter.

CXL 4.0 laid the groundwork. The CXL Consortium’s documentation explicitly states that “optical connectivity is obtained through retimers.” Four-retimer support extends physical reach. Read this as the starting point for multi-rack CXL configurations and, further out, optical CXL fabric. The spec including optical support and actual commercial rack-scale disaggregation are two different things, but the direction is clear.

9. Will CXL Actually Be Adopted?

By this point, CXL’s value proposition should be clear.

But there is an honest question that still needs to be asked. The problem CXL is trying to solve, building a larger and cheaper memory tier outside HBM, is not uniquely CXL’s to solve. Other technologies are approaching the same problem differently, and some approaches involve no new hardware at all.

The most direct overlap. SanDisk’s first-generation HBF (High Bandwidth Flash) targets 1.6 TB/s read bandwidth, 512GB capacity with a 16-die stack, and an HBM4-comparable footprint. The idea is to stack NAND flash via TSV to place much higher capacity near the GPU at far lower cost per bit than HBM. SK Hynix proposed the H3 architecture combining HBM and HBF, and in February 2026 began an HBF standardization workstream under OCP alongside SanDisk.

This is a meaningful challenge to CXL’s KV cache story. Instead of offloading KV cache to an external memory tier over PCIe and a CXL switch, put an additional memory tier right next to the GPU. From the GPU’s perspective, a nearby memory tier within the same package neighborhood is more natural than accessing external memory through a switch.

That said, HBF is fundamentally an on-package memory expansion play. CXL is a host-attached coherent external memory fabric. HBF is compelling for GPU-centric inference tiers, but it does not address server-to-server memory pooling, DDR reuse, shared memory shelves, or rack-scale disaggregation. NAND-specific challenges around write characteristics, endurance, controller complexity, and package integration remain. HBF threatens one part of CXL’s narrative, specifically the KV cache offload story, but not the whole thing.

LPDDR-Based Node-Local Expansion

Micron’s SOCAMM2 is the representative example. It directly positions itself around offloading KV cache from HBM to high-capacity LPDDR, with Micron’s own data showing a 2.3x improvement in time to first token for inference workloads. The key distinction from CXL: this adds a cheaper, larger memory tier inside the server without separating memory from the rack. Architecturally less ambitious, but far simpler for the customer to adopt. Some portion of CXL’s potential demand could get absorbed by LPDDR-based local expansion first.

MRDIMM

MRDIMM (Multiplexed Rank DIMM) doubles DDR5 bandwidth as a drop-in replacement for existing DIMM slots. Gen1 at 8,800 MT/s is already shipping on Intel Granite Rapids, with Gen2 at 12,800 MT/s arriving in 2026. For customers who want higher bandwidth per core and higher capacity per core within the same server rather than rack-scale memory pooling, MRDIMM is a much more conservative and easier-to-deploy solution. Some of what CXL is trying to sell can be addressed with MRDIMM before CXL even arrives.

Flash as a Memory Tier

HBF targets on-package flash. More broadly, GPU-accessible flash and storage-class expansion represent a related direction. None of these provide coherent memory semantics the way CXL does, but the core question for inference infrastructure is shifting. It is less about whether something is DRAM or storage, and more about how cheaply, how large, and at how predictable a latency the working set outside HBM can be placed. For workloads with relatively predictable access patterns, like KV cache, cold tensors, and prefetchable context data, flash-based tiers could establish a position faster than expected.

Software

Often underestimated, but genuinely important. Part of CXL’s appeal is that it touches real problems like KV cache management and external memory tiering directly. But software can move into that space first. Techniques like cache eviction strategies, prefix reuse, paged attention, KV compression, and remote KV cache sharing continue to advance. The more effective software optimization becomes, the fewer situations require hardware-level external memory expansion. For customers, continuing to run existing GPUs and servers while optimizing in software is a much lower-friction path than waiting for a new hardware ecosystem to mature. CXL is competing not just against other hardware technologies, but against the entire category of “ways to mitigate the problem without buying new hardware.”

But These Approaches Share a Common Limitation

Look at all five approaches again and one pattern stands out. Every single one stops at the server boundary.

HBF attaches flash within a GPU’s package. The GPU next to it cannot access that memory. SOCAMM2 adds an LPDDR tier inside a server. Not shareable across servers. MRDIMM increases DDR5 bandwidth within a server. Nothing goes to another server. Flash tiers are server-internal expansions. Software optimization works within the existing hardware envelope.

Only CXL crosses server boundaries. Multiple servers sharing one memory pool dynamically, allocating and reclaiming based on workload, mixing DDR4 and DDR5 across generations. This is structurally impossible with node-local technologies.

Beyond crossing server boundaries, there are technical advantages that belong to CXL alone.

It is already in every server. From Intel Sapphire Rapids through Granite Rapids, from AMD EPYC Genoa through Turin, CXL ports are standard on currently shipping server CPUs. Add a memory expander and you are running. HBF requires GPU package redesign. SOCAMM2 requires server board redesign. “Already deployed” is an overwhelming adoption barrier advantage.

It runs on existing PCIe infrastructure. No new cables, connectors, or physical infrastructure required. The PCIe slots already in the server are all you need. HBF requires new interposer designs and packaging processes. CXL adds a memory tier on top of existing infrastructure.

It is memory-agnostic. Plug DDR4 behind a CXL controller. Plug DDR5. Plug flash later. The memory hardware is decoupled from the protocol. HBF is tied to NAND. MRDIMM is tied to DDR5. With CXL, whatever goes behind the controller presents the same coherent memory semantics to the host.

It is the only open standard providing cache coherency. HBF, SOCAMM2, MRDIMM, and flash tiers do not provide cache coherency. A single host can expand its local memory, but multiple hosts simultaneously sharing the same memory with data consistency maintained requires CXL. There is no alternative.

The Argument That Is Stronger Than Technical Specs

Technical advantages alone do not guarantee that a standard survives. History has more examples of inferior standards surviving on ecosystem momentum than superior standards winning on merit.

CXL’s real survival argument is not in the spec sheet. It is in who controls the capital.

Hyperscalers want CXL.

Memory pooling reduces DRAM procurement. Less stranding means less new DRAM to buy. DDR4 reuse means less new DDR5 to buy. At hyperscaler scale, these translate to hundreds of millions of dollars in annual savings. CXL also increases hyperscaler leverage over memory vendors. When memory is locked to individual servers, vendors can raise prices and customers pay. Pooling reduces total demand and improves negotiating position. Google, Microsoft, Meta, and Alibaba are founding members of the CXL Consortium. That is not a coincidence.

Memory vendors are in a complicated position.

Broad CXL adoption could reduce total DRAM demand. Higher utilization means the same workloads can run on less DRAM. That is unwelcome. But CXL memory modules are also a new product category, and not participating means letting a competitor take the market. This is where HBF’s political dimension becomes visible. SK Hynix and SanDisk pushing HBF hard is not purely technical enthusiasm. HBF means “buy new NAND.” That requires new purchases. CXL pooling means “use existing DRAM more efficiently.” That reduces new purchases. For memory vendors, HBF is a better revenue story than CXL.

But the companies controlling AI infrastructure capex are the hyperscalers.

Google and AWS alone have guided to nearly $400B in combined capex for 2026. Whoever controls where that money flows determines what technologies win. When Microsoft deploys CXL in Azure, it is not just a technical decision. It is a market signal. Technology survives not because it is technically superior, but because the organizations spending the money have decided to use it.

The Remaining Risks

Timeline: CXL 4.0 multi-rack systems are a 2027 story at the earliest. A spec being finalized and production hardware shipping are entirely different milestones.

Software maturity: CXL memory appears to the OS as a separate NUMA node. Getting real value out of it requires NUMA-aware applications or middleware like MemVerge Memory Machine. The ecosystem is still early.

Competing technology timelines: HBF samples are expected in the second half of 2026. MRDIMM Gen2 arrives in 2026. Software optimization is progressing right now. If these technologies adequately solve memory problems at the node-local level before CXL’s multi-rack deployment arrives in 2027, CXL could find that the problem it was designed to solve has already been partially solved. For CXL to matter, it needs to prove its value in memory pooling, DDR reuse, and disaggregated architectures, not just KV cache offload.

Conclusion

CXL is not the primary AI datacenter interconnect. It will not become one. GPU-to-GPU connectivity belongs to NVLink. Node-to-node networking belongs to Ethernet. CXL operates between those layers, reshaping the memory tier. It is not a glamorous role. But the amount of money moving around this unglamorous technology is large.

There are three reasons to pay attention to CXL right now.

First, the economics of memory are changing. Memory is the single most expensive line item in AI infrastructure, and 30% of it is being wasted. CXL is the only open standard that addresses this waste directly, and the organizations that want it most are the hyperscalers spending hundreds of billions of dollars in capex every year. Technologies that the biggest spenders want tend to survive.

Second, the combination with optics has begun. CXL confined to a single server tops out as a memory expander. CXL running over optics enables an architecture where the entire datacenter’s memory becomes one unified pool. Companies are already placing billion-dollar bets at this intersection. If you have been watching the optics theme without tracking the protocol running on top of those optical links, you have been missing half the picture.

Third, the alternatives cannot replace all of CXL. HBF, MRDIMM, LPDDR, and software optimization can absorb portions of CXL’s use cases. But cross-server memory pooling, DDR reuse, and cache-coherent disaggregation are structurally outside what those technologies can do. CXL’s exclusive territory is narrow, but nothing else occupies it.

The risks are real. Multi-rack CXL is a 2027 story. The software ecosystem is still early. There is a genuine possibility that alternative technologies mitigate the memory problem well enough before CXL’s full vision arrives. None of this is guaranteed to play out as the roadmaps suggest. But CXL ports are already standard on every server shipping today. NVIDIA has adopted it as a complement to its own stack. Microsoft and Google have started committing real capital.

That combination is hard to ignore, and it is reason enough to start tracking where this technology goes.
